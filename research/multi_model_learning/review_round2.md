3. **Prior art.** This is not remotely blank territory. Intermediate/layer distillation is old: [FitNets](https://arxiv.org/abs/1412.6550), [Patient KD](https://aclanthology.org/D19-1441/), [TinyBERT](https://aclanthology.org/2020.findings-emnlp.372/), [MiniLMv2](https://aclanthology.org/2021.findings-acl.188/). Representation matching and alignment are old: [CKA](https://proceedings.mlr.press/v97/kornblith19a.html), [SVCCA](https://arxiv.org/abs/1706.05806), [data2vec](https://proceedings.mlr.press/v162/baevski22a.html). Dynamic multi-teacher selection already exists: [Reinforced Multi-Teacher Selection](https://arxiv.org/abs/2012.06048). Cross-tokenizer KD is active prior art, not novel: [Dual-Space KD](https://aclanthology.org/2024.emnlp-main.1010/) and [Contextual Dynamic Mapping](https://aclanthology.org/2025.findings-acl.419.pdf). Offline pretraining-time KD across families also exists: [MiniPLM](https://arxiv.org/abs/2410.17215). The actual novelty here is much narrower: stage-contract distillation inside Sutra’s stage-superposition recurrent system, gated by the same residual-gain law as compute control, and validated by teacher-free consolidation. Fix: cut all broad novelty claims and state only that narrower integration claim.

4. **Feasibility of `dim=128` CPU probes.** They are too small to validate the thesis. The repo already has a negative small-scale result at [RESEARCH.md:1815](/C:/Users/devan/OneDrive/Desktop/Projects/AI%20Moonshots/moonshot-sutra/research/RESEARCH.md#L1815), and the same file repeatedly says `dim=128` is triage only; the multi-model summary itself admits that in [RESEARCH.md:5977](/C:/Users/devan/OneDrive/Desktop/Projects/AI%20Moonshots/moonshot-sutra/research/RESEARCH.md#L5977). At that scale, stage semantics, alignment scans, and routing/memory geometry are too immature to support a strong thesis test. Fix: treat `128` only as a kill-screen for wiring bugs and obvious negative transfer; require the first real validation at `768` after separation is demonstrated, and require `1024` plus consolidation before any claim of success.

5. **Risks.** Worst case, this actively hurts Sutra. It can confound the current science question, mask whether attached-history actually creates separation, import incompatible teacher biases into different stages, and turn the system into a fixed-pass imitator instead of a coherent state machine. The repo itself says the current bottleneck is “a dense fixed-pass updater pretending to be a state machine” at [RESEARCH.md:5890](/C:/Users/devan/OneDrive/Desktop/Projects/AI%20Moonshots/moonshot-sutra/research/RESEARCH.md#L5890); adding more teacher losses before solving that can make the wrong system look better. Fix: forbid teacher losses until the elastic-compute substrate is real, and require comparison against “same compute spent on more data/self-training” so teacher gains are not a mirage.

6. **Simplification.** The current design is overengineered. The absolute minimal test is: one teacher, one stage, one tokenizer regime, one insertion point, hard-token slices only, late passes only, zero-gated path, mandatory teacher dropout/consolidation. Concretely: Stage 7 only, same-tokenizer AR teacher if possible, top-K logit transfer only on hard late-pass tokens, with success defined as retained teacher-free gain after annealing. Do not start with CKA + SVCCA + cross-tokenizer span alignment + Stage 4/5 geometry + multi-teacher selection all at once.
