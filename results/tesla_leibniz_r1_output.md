- The trainer uses `BATCH_SIZE=4`, `GRAD_ACCUM=16`, `SEQ_LEN=512`, so each optimizer step is `32,768` tokens. Step `11800` therefore means about `386.7M` tokens consumed by this run. The repo’s `20.72B` number is corpus size, not consumed tokens.
- The most important measurement correction in the repo is that sampled intermediate CE inverted the true late-pass story. At the same time, collapse probes showed passes `0-9` are near-fixed-point and much of the useful work is deferred into the tail. The right conclusion is not “late passes are destructive.” The right conclusion is “late passes are constructive, but the current training dynamics make the model postpone too much work until very late.”

## 1. Assumption Challenges With Both-Sides Analysis

**Assumption: `12` fixed recurrent passes is the right depth.**
- For keeping it: full-vocab truncation on the current checkpoint showed `12` passes massively outperform truncated versions, with `12` passes at about `7.59` BPT and `10` passes still disastrously worse. That means the existing model is genuinely using late recurrence for real quality, so blindly shrinking depth would amputate working computation.
- Against keeping it: collapse metrics at step `10K` showed passes `0-9` are extremely similar and the final pass performs a huge rewrite. That means the current system is not using `12` passes as `12` distinct refinements. It is using many near-idle passes plus one or two decisive tail passes. Fixed `12` therefore looks more like a training artifact than an optimal compute policy.
- Evidence that resolves it: train a simplified recurrent canary with identical core and exact-memory path, but sample depth during training and compare `D=1` vs `D in {1,2,3,4}` at matched average compute. If `D>1` wins after depth is forced to matter earlier, recurrence is real. If not, the current `12`-pass gain is mostly compensation for bad optimization.
- Lean: keep recurrence alive, reject fixed `12` as a design truth. Confidence `8/10` because full-vocab eval proves late recurrence matters, but collapse data proves the current depth allocation is wrong.

**Assumption: pure weight sharing across passes is the right parameter strategy.**
- For keeping it: with one laptop GPU, sharing is the only reason deep recurrence is affordable at all. Shared weights also fit the project’s platform goal, because a strong reusable core is easier to improve than `12` separate blocks.
- Against keeping it: the repo’s own gradient analysis shows `L_final` dominates pass gradients, so fully shared passes are incentivized to converge toward one final-pass-like behavior. Without pass identity, shared recurrence invites fixed-point collapse. Most successful recursive models in the literature use a strong shared core plus lightweight pass-specific adapters, not total sharing with no pass identity.
- Evidence that resolves it: compare three matched models on the same simplified core: fully shared, shared plus pass-conditioned normalization, and shared plus very small per-pass LoRA-style adapters. Measure pass cosine spread, passwise BPT contributions, and final BPT.
- Lean: keep a mostly shared core, but add lightweight pass identity. Confidence `8/10` because compute constraints strongly favor sharing, yet the current collapse is also strong evidence that pure sharing is too rigid.

**Assumption: the `7`-stage decomposition is the right abstraction.**
- For keeping it: the stage framework already helped find and reason about behavior. The repo shows prose and code follow different stage paths, and the modular story is central to Outcomes 2 and 3.
- Against keeping it: the current code does not actually realize a clean modular ABI. Stage outputs are blended, not exposed as stable contracts. Two stages are mostly dead because tokenization already pre-solves them. The decomposition is conceptually useful but operationally overstated.
- Evidence that resolves it: compare current `7`-stage heavy bank against a single shared content block with lightweight stage-conditioned control, then test whether capability-specific probes are easier to diagnose or improve in the lighter design.
- Lean: keep stages as control semantics, not as `7` heavyweight transform banks. Confidence `9/10` because the conceptual decomposition helps reasoning, but the current heavy implementation has not earned its cost.

**Assumption: the current `StageBank` mechanism, meaning `7` separate FFNs with top-2 stage sparsity, is the right implementation of stage specialization.**
- For keeping it: distinct per-stage transforms are the cleanest way to let different processing modes do genuinely different work, and top-2 projection is at least an attempt to bound compute.
- Against keeping it: in the current implementation, if any token activates a stage, that stage MLP runs over the full tensor. So the advertised token-conditional cheapness is weaker than the design story. More importantly, `StageBank` consumes about `16.5M` parameters, which is a huge share of the non-embedding budget, and that budget is already scarce.
- Evidence that resolves it: build a single shared SwiGLU or SiLU content block with `pi`-conditioned scale/shift or gating, keep router/writer/scratchpad unchanged, and compare compute-normalized results. If it matches or beats the stage bank, the stage bank is unjustified at this scale.
- Lean: replace `StageBank` with one strong shared block plus lightweight control. Confidence `9/10` because the current code-grounded compute and parameter costs are too high for the amount of validated benefit.

**Assumption: the causal scratchpad is load-bearing and should survive simplification.**
- For keeping it: scratchpad is the most consistent positive architectural signal in the repo. It improved BPT in Chrome, its benefit scaled up rather than disappearing, and it is the clearest answer to recurrent exact-recall weakness. The current implementation is also relatively simple: shared slots, causal prefix summaries, and gated EMA writes.
- Against keeping it: the current scratchpad is still crude. Eight shared slots may be too small, writes can smear incompatible facts, and reads are soft attention over shared slots rather than exact key-value retrieval. It may be helping optimization without yet solving exact recall robustly.
- Evidence that resolves it: run controlled long-range retrieval and exact-copy probes with scratchpad on, read-disabled, write-disabled, and fully removed, while also measuring held-out BPT. If recall quality collapses specifically when scratchpad is disabled, it is truly load-bearing rather than merely regularizing.
- Lean: keep exact-memory support as non-negotiable, but treat the current scratchpad as a first working draft rather than the final memory design. Confidence `8/10` because the empirical signal is the strongest in the repo, but the mechanism still needs sharper addressing.

**Assumption: `BayesianWrite` and `lambda` are the right story for state updates and future halting.**
- For keeping it: the write mechanism is clearly central, and the gain clamp fixed a real NaN failure. Precision-weighted updates also give a principled way to prevent uncontrolled state overwriting.
- Against keeping it: the repo’s own production probe showed `lambda` failed as a correctness or halting signal and was even anti-correlated with correctness. The current writer may still be useful, but the “Bayesian precision equals confidence and halting” interpretation is not yet earned. Monotone precision also makes contradiction-driven revision difficult.
- Evidence that resolves it: compare three writers on the same simplified recurrent core: current `BayesianWrite`, a simpler bounded gated residual writer, and a conflict-aware writer that can lower certainty on contradictory evidence. Evaluate BPT, stability, revision behavior, and calibration of the update signal.
- Lean: keep a bounded write mechanism, demote the strong `lambda` confidence story. Confidence `8/10` because the write path is probably real, but the current theory wrapped around it is too strong for the evidence.

**Assumption: the current cross-position routing story is correct.**
- For keeping it: routing is clearly important. Earlier ablations showed removing router hurt badly, and code-like content takes more routing-heavy paths than prose. Some form of local plus sparse long-range communication is mandatory if recurrence is to work.
- Against keeping it: the current router still forms a full score matrix before sparsification, so the O(n) efficiency story is not yet true in implementation. It also uses the same content state for both semantics and routing, which may entangle control and content too early.
- Evidence that resolves it: build a true sparse or two-stage router that first proposes a tiny candidate set cheaply, then scores only those candidates exactly, and compare it against the current router on held-out LM quality plus exact recall tasks.
- Lean: keep routing and exact recall, but simplify the mechanism and separate content from routing control more aggressively. Confidence `7/10` because routing is clearly needed, but the current implementation has not achieved the intended efficiency.

**Assumption: pheromone or delayed trace routing is worth keeping.**
- For keeping it: some earlier Chrome work showed late-step improvements, and the idea is philosophically aligned with stigmergic shared-medium computation.
- Against keeping it: the evidence is mixed, the latest internal design direction already wants it removed, and positive feedback on an attractor-prone recurrent system is exactly the wrong failure mode to amplify. Since scratchpad already provides a shared medium, pheromone looks like extra complexity before the core recurrence gate is won.
- Evidence that resolves it: after metric cleanup and simplification, run a clean ablation with and without pheromone on the same shared-core recurrent canary. If it does not improve fixed-slice BPT and generation while preserving collapse metrics, it should stay out.
- Lean: remove pheromone from the next core design. Confidence `7/10` because the philosophical appeal is high but the empirical case is weak relative to the extra moving parts.

**Assumption: the current training recipe, meaning fixed depth plus `L_final + weak sampled L_step + L_probe`, is basically right.**
- For keeping part of it: attached history matters, and the sampled `L_step` sits in a “Goldilocks” zone. The repo showed no `L_step` makes late-pass usefulness decay, while exact full-vocab intermediate CE catastrophically collapses late-pass contribution by forcing intermediate passes to become premature output heads. So a weak intermediate objective is helping even if it is imperfect.
- Against keeping the full recipe: fixed-depth training is almost certainly the root of deferred final-pass computation, because the gradient structure tells every pass to optimize for the final pass. The sampled CE signal is also biased and can invert the apparent per-pass quality ranking, so it is a dangerous metric even if its gradients are pragmatically useful.
- Evidence that resolves it: keep weak `L_step`, but change the gradient structure first with random-depth or progressive-depth training. Then remeasure pass contribution distribution and generation quality on a fixed eval slice.
- Lean: keep weak intermediate supervision, but change the depth training regime before inventing more controller heads. Confidence `9/10` because the repo already contains a decisive three-way comparison: full exact intermediate CE is too strong, none is too weak, the weak proxy is helpful but should not stay inside a fixed-depth regime forever.

**Assumption: training from scratch is an acceptable medium-term strategy.**
- For keeping it temporarily: it is cleaner for diagnosis, avoids transfer mismatches while architecture is still moving, and earlier broad teacher mixing did not help at tiny scale.
- Against keeping it as a principle: it directly weakens Outcome 4. With only one GPU and under a billion consumed tokens so far, from-scratch training is a self-imposed handicap against models trained on hundreds of billions or trillions of tokens. The mission explicitly requires learning from existing knowledge, not just raw text.
- Evidence that resolves it: once the recurrence gate is cleaner, test one narrow data-efficiency lever at production scale, not a bundle. The cleanest candidates are stage-3-style local pretraining from scratch or one stage-specific teacher on hard-token slices only.
- Lean: treat from-scratch as a diagnostic convenience, not as a design commitment. Confidence `8/10` because it is sensible for debugging, but it is incompatible with the long-term mission.

**Assumption: the current `50K` GPT-2 tokenizer is the right tokenization choice.**
- For keeping it now: it avoids destructive checkpoint surgery, gives stable tooling, and the recurrent architecture changes the usual tokenizer tradeoff enough that a rushed tokenizer change could create more confusion than insight.
- Against keeping it long term: embeddings plus positional embeddings consume about `40.2M` parameters, which is about `58.8%` of the whole model. That leaves only about `28.2M` parameters for the actual recurrent machinery. Tokenizer choice is therefore not cosmetic here; it is stealing the majority of the budget from computation.
- Evidence that resolves it: after the recurrence gate is passed, run one late-stage tokenizer study on the simplified core using either a smaller vocab or a hierarchical bridge, and evaluate total-quality-per-total-parameter, not just raw BPT.
- Lean: do not change tokenizer before the recurrence gate, but assume tokenizer redesign is likely necessary later. Confidence `7/10` because the embedding tax is obviously large, but changing tokenizer too early would confound more important questions.

**Assumption: the recurrence thesis is already justified at `68M`.**
- For keeping it alive: the full-vocab results prove late passes matter, scratchpad helps materially, content-dependent transitions are real, and external recursive models like ITT and MoR show small-to-medium recurrence can work when the recipe is right. So recurrence is not a dead idea.
- Against claiming success now: the current model is still not genuinely intelligent by the project’s own standard. Generation is repetitive, benchmark evidence is near floor, the active run has only consumed about `0.387B` tokens, and the architecture has not yet passed the crucial compute-normalized test that `D>1` beats `D=1` on the same core. At this scale, recurrence is still a hypothesis, not a win.
- Evidence that resolves it: one decisive gate on a simplified exact-memory recurrent core, with `D=1` as the internal control and `D>1` trained under random depth at matched compute.
- Lean: recurrence is viable but unproven; current evidence justifies a cleaner test, not a full doctrine. Confidence `8/10` because there is enough positive signal to continue, but not enough to freeze the roadmap.

**Assumption: the platform and ABI story is already real enough to guide design.**
- For keeping it as a goal: the repo already benefits from modular thinking. Bugs were traced to concrete components and individual mechanisms can be reasoned about separately.
- Against claiming it now: current stage contracts are not stable enough for outside composition. Outputs are blended, modules are not yet exposed as real contracts, and the design is still changing too fast. Freezing an ABI now would mostly fossilize the wrong abstractions.
- Evidence that resolves it: after simplification, expose explicit stage or module contracts, then show that one component can be improved in isolation and compose back into the system without full retraining.
- Lean: keep democratized modularity as a destination, not as a present-tense claim. Confidence `8/10` because the direction is correct, but the present implementation is not yet community-grade infrastructure.

## 2. Research Requests For Claude To Execute

- I need a narrow literature map of successful recursive or shared-weight language models in the `100M-350M` range, with exact answers to four questions: how much of the model is shared, what pass identity mechanism is used, how depth is sampled during training, and whether exact memory is present. I do not need a broad survey. I need the concrete recipes that actually worked.
- I need a focused survey of representation-level intermediate objectives for recurrent or iterative models. The key question is not “what predicts tokens early?” but “what keeps intermediate states useful for later passes without forcing them to become premature output heads?” Contrastive, predictive, consistency, or mutual-information-style objectives are relevant only if they have causal LM or iterative-model evidence.
- I need a rigorous survey of function-preserving growth and warm-start methods that survive meaningful architectural edits in causal LMs. I care about additive adapters, zero-influence birth, reversible migrations, and any evidence that tokenizer or routing changes can be introduced without throwing away already-paid-for knowledge.
- I need a compact survey of exact-recall mechanisms compatible with recurrent models on one GPU. I do not want generic long-context methods. I want mechanisms that preserve exact or near-exact recall without collapsing into full dense attention.
- I need a targeted survey of tokenizer and embedding-tax tradeoffs for sub-`200M` models, especially when recurrence multiplies compute but embeddings consume most parameters. The key question is whether smaller or hierarchical vocabularies improve total intelligence per total parameter once recurrent depth is in the picture.

## 3. Experiment/Probe Requests With Methodology For Claude To Execute

- **Probe A: simplified recurrence gate, run when the GPU is free.** Method: implement one strong shared content block plus existing-style router, bounded writer, and scratchpad, with no `7`-FFN stage bank and no pheromone. Train two arms under matched average compute: `D=1` and `D in {1,2,3,4}` with random-depth sampling. Use the same tokenizer and data mix. Evaluate full-vocab BPT, fixed-slice generation diversity, and one exact-recall task. Expected outcome: if recurrence is genuinely valuable at this scale, `D>1` should beat `D=1` on compute-normalized learning and recall. What it tells us: whether deep recurrence is the right organizing principle or merely compensation for a weak core.
- **Probe B: random-depth on the current `v0.6.0a` scaffold, run when the GPU is free.** Method: minimal code change only. Sample training depth from a deep-biased distribution, keep attached history, keep the current weak sampled `L_step`, and compare against fixed `12` passes from the same checkpoint family. Log full-vocab per-pass BPT, pass cosine, lambda profile, and generation metrics on a fixed eval slice. Expected outcome: earlier passes should become more useful and the final-pass rewrite should be less extreme. What it tells us: whether gradient-structure repair alone can fix deferred computation before heavier architecture changes.
- **Probe C: `7`-bank versus shared-core canary, run when the GPU is free.** Method: matched parameter and matched compute comparison between the current `StageBank` style and a single shared block with `pi`-conditioned modulation. Keep router, writer, scratchpad, tokenizer, and training recipe otherwise aligned. Expected outcome: if the shared core wins or ties, the heavy stage bank is not justified at `68M`. What it tells us: whether stage specialization should live in control or in expensive duplicate transforms.
- **Probe D: scratchpad load-bearing audit, CPU first and GPU only if needed.** Method: take the latest checkpoint and run fixed-slice evaluation in four modes: full system, scratchpad read disabled, scratchpad write disabled, and scratchpad removed. Add one synthetic long-range recall task and one in-domain reference-resolution test. Expected outcome: if scratchpad is truly sacred, removing it should sharply hurt exact recall and long-range coherence even when BPT changes only modestly. What it tells us: whether exact memory is a real pillar or just an optimization helper.
- **Probe E: pass disagreement as intrinsic uncertainty, CPU now.** Method: on the latest checkpoint, compute per-token disagreement features across passes using full-vocab quantities, not sampled CE: entropy spread, logit KL spread, passwise CE spread, and state cosine spread. Correlate them with future-gain, hard-token classes, and repetition onset on fixed cached batches. Expected outcome: a positive correlation would validate disagreement as an internal training-weighting and controller signal. What it tells us: whether active compression can be driven by the model’s own recurrent disagreement rather than an external teacher.
- **Probe F: one narrow data-efficiency lever at production scale, only after the recurrence gate is cleaner.** Method: choose exactly one of these, not both: local masked-reconstruction pretraining before LM training, or one stage-specific teacher on hard-token late-pass slices. Run it on the simplified recurrent core at `768` first. Expected outcome: a meaningful BPT-per-token improvement without destroying late-pass behavior. What it tells us: whether Outcome 4 has a practical near-term lever or remains mostly aspirational.

## 4. Per-Outcome Confidence (1-10) With Justification

- **Outcome 1 (Intelligence): `4/10`** - The project has real signs of life: full-vocab measurements show late recurrence is doing useful work, scratchpad materially helps, and the system can already learn nontrivial language structure. But the active model still produces repetitive generation, benchmark evidence remains near floor, and the core recurrence gate has not yet shown compute-normalized superiority over a shallow control. The evidence supports “promising mechanism incubator,” not “intelligence achieved.”
- **Outcome 2 (Improvability): `6/10`** - The project is unusually good at discovering its own failure modes. The repo already isolated a causality leak, a NaN root cause, a detached-history bug, and a metric inversion bug. That is exactly the kind of debug visibility Outcome 2 wants. The score is not higher because the current `7`-stage implementation still overstates its own modularity; the ABI is not yet real enough for surgical community improvement.
- **Outcome 3 (Democratization): `4/10`** - The philosophy is strong and the architecture is trying to move in the right direction, but the current system is not yet stable enough for true external composition. Frequent architecture changes, absent stage contracts, and weak function-preserving growth mean an outside contributor would still be contributing into moving sand. The idea is credible. The present implementation is not.
- **Outcome 4 (Data Efficiency): `3/10`** - The mission says “learn from everything,” but the active system is still mostly from-scratch next-token training. There are hints of future levers, such as narrow teacher use, NCA-style local pretraining, and disagreement-driven weighting, but almost none of that is production-validated at the relevant scale. The architecture may become data-efficient, but the repo does not yet prove it.
- **Outcome 5 (Inference Efficiency): `2/10`** - Today’s system is a dense fixed-`12`-pass recurrent model with no acting halting, no token retirement, and no realized average-depth savings. The live run still spends most tokens through the full loop, and collapse evidence suggests much of that compute is poorly allocated. The long-term thesis remains plausible, but the current implementation is far from an efficient inference story.

## 5. What Would Raise My Confidence For Each Outcome

- **Outcome 1 (Intelligence)** - Confidence rises if a simplified recurrent core with exact memory shows `D>1` beating `D=1` at matched compute on full-vocab BPT, fixed-slice generation, and exact recall. Confidence also rises if the current run’s generation diversity improves materially by `20K` steps without needing heavy decode hacks.
- **Outcome 2 (Improvability)** - Confidence rises if one module can be upgraded in isolation on the simplified core and the resulting benefit stays localized and composable. Confidence also rises if per-module metrics predict user-visible behavior changes instead of only internal proxy shifts.
- **Outcome 3 (Democratization)** - Confidence rises if one major version transition preserves behavior at zero gate, meaning a new module can be born with zero influence and then earn its way into the model. Confidence also rises if a stable contract for memory or writing can be exposed and improved without retraining the whole system.
- **Outcome 4 (Data Efficiency)** - Confidence rises if one narrow data-efficiency intervention at `768` or `135M` gives a clear tokens-to-target improvement. Confidence also rises if pass disagreement proves to be a calibrated internal difficulty signal that improves learning curve slope.
- **Outcome 5 (Inference Efficiency)** - Confidence rises if random-depth or token retirement cuts average effective depth without quality loss, and if exact recall remains available through a cheap side path rather than forcing dense recurrent compute on every token.

## 6. What Would Lower My Confidence For Each Outcome

- **Outcome 1 (Intelligence)** - Confidence falls if the simplified recurrence gate shows `D>1` does not beat `D=1` on compute-normalized quality. Confidence also falls if generation remains repetition-dominated after the model has consumed roughly Chinchilla-scale tokens for its size.
- **Outcome 2 (Improvability)** - Confidence falls if simplification reveals that almost every useful gain comes from one opaque shared block and cannot be cleanly localized. Confidence also falls if every improvement still requires full retraining of the whole system.
- **Outcome 3 (Democratization)** - Confidence falls if major version changes keep breaking checkpoint continuity or tokenizer compatibility in ways that force fresh training. Confidence also falls if “stage” remains mostly rhetorical and never becomes a real external contract.
- **Outcome 4 (Data Efficiency)** - Confidence falls if every teacher, curriculum, or local-pretraining attempt at relevant scale either fails or only matches the gain from simply feeding more raw tokens. Confidence also falls if the architecture never learns better than standard models once token count is normalized.
- **Outcome 5 (Inference Efficiency)** - Confidence falls if recurrence only works when nearly all tokens use near-max depth, or if exact recall requires routing mechanisms so dense that they erase the compute advantage. Confidence also falls if halting signals remain miscalibrated even after the recurrent core is repaired.

## 7. Need More Data With Specific Asks

I do **not** recommend freezing a full all-outcomes architecture yet. The missing evidence is too central. I need five things before I would lock a real design: `1)` the simplified `D>1` versus `D=1` recurrence gate, `2)` a shared-core versus `7`-bank comparison, `3)` a scratchpad load-bearing audit on exact recall, `4)` a calibration study for pass disagreement as intrinsic uncertainty, and `5)` one validated data-efficiency lever at production-relevant scale.

The best next design hypothesis is clear even though the full design is not. Test a **minimal shared-core recurrent model** with one strong shared content block, local plus truly sparse routing, an exact-memory scratchpad path, a bounded writer, lightweight pass identity, random-depth training, and the current weak intermediate supervision. Do **not** carry forward the heavy `7`-FFN stage bank, do **not** treat fixed `12` passes as sacred, do **not** rely on `lambda` as the halting story, and do **not** add more control machinery before the core recurrence gate is won. If that simplified recurrent core works, the path to all five outcomes becomes much clearer. If it does not, the mission should continue through a shallower exact-memory hybrid rather than deeper ideological recurrence.
