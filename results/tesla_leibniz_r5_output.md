1. **Assumption challenges with both-sides analysis**
- **Pass count (`12` passes) is a real capability, but not yet a real architecture.** For: the truncation sweep was catastrophic from `12 -> 10` passes (`7.59 -> 18.39` BPT at 8.3K), and the final pass currently carries real quality. Against: pass dynamics at 14K and 20K show passes `0-10/11` are near-identity drift with entropy near-uniform until the last pass, so `12` is currently a training workaround, not an earned decomposition. Obvious interpretation: recurrence is broken. Alternatives: the loss geometry is broken; shared weights force a latent/decode phase split; the front-end is so bloated that only the last pass has enough effective capacity left to decode. Resolve with `rd12`, then a matched-compute `D=1` vs `RD4` vs `RD8` gate on a simplified shared-core family. Lean: keep `12` only as the warm-start teacher depth, target `Dmax=8` once collapse is removed. Confidence: `8/10`.
- **Full weight sharing should survive, but only if pass identity becomes explicit.** For: sharing is the only reason recurrence fits the 68M budget, and it preserves warm-start compatibility. Against: the near-orthogonal final pass strongly suggests the same block is being asked to be both encoder and decoder with no conditioning, which is exactly the failure mode pass-conditioned recurrence papers predict. Obvious interpretation: weight sharing itself is the problem. Alternatives: unconditioned sharing is the problem; the controller is missing; the final loss makes one “decode” behavior globally optimal. Resolve by comparing `rd12` alone vs `rd12 + pass conditioning` vs `rd12 + cheap per-pass adapters`. Lean: keep one shared content core, add cheap pass-conditioned control. Confidence: `9/10`.
- **Seven named stages improved interpretability, but seven content banks is the wrong allocation at this scale.** For: stage-bank diversity is real, with mean off-diagonal cosine around `0.012`, so the system did learn differentiated transforms. Against: stages `0-1` are barely trained, and too much parameter budget is trapped in underused branches. Obvious interpretation: stages are fake. Alternatives: the decomposition is useful but over-granular; the stage labels are right but the content should be shared; early stages are starved by collapse rather than conceptually invalid. Resolve by measuring MI between current `pi` histories and a 4-mode pseudo-label map, then comparing 7-bank vs 4-mode control on BPT, generation, and token-type recall. Lean: collapse the explicit control interface to `compose / route / retrieve / verify`, keep modularity at the controller/memory ABI rather than in seven separate MLP banks. Confidence: `8/10`.
- **The StageBank mechanism has earned “specialization exists,” not “specialized content banks are optimal.”** For: orthogonal banks prove distinct transforms can emerge. Against: separate banks duplicate content capacity and waste parameters on little-used modes; a shared SwiGLU core with control embeddings is a better fit for Outcomes 1, 2, and 3. Resolve by a gated parallel transfer: train a shared core to mimic the live bank, ramp it up, and delete the bank only if BPT, throughput, and generation hold. Lean: replace the StageBank with one shared content core plus cheap control. Confidence: `8/10`.
- **The scratchpad is load-bearing as workspace, but not as precise memory.** For: it gave real BPT lift and is a plausible discourse/workspace mechanism. Against: the exact failure buckets are numbers, proper nouns, and acronyms, which are exactly where an 8-slot EMA workspace should fail. Obvious interpretation: memory is not the bottleneck. Alternatives: memory is the bottleneck but the current memory type is wrong; memory demand is hidden by pass collapse; scratchpad helps fluency while exact recall is still missing. Resolve with a memory-only swap after collapse repair and controller routing. Lean: keep scratchpad as diffuse workspace only. Confidence: `10/10`.
- **The kNN result does not refute retrieval; it refutes post-hoc retrieval on weak representations.** For keeping retrieval: the datastore was tiny, the model is small, and kNN-LM is known to help factual/rare-token PPL more than open-ended generation. Against: if a runtime memory cannot lift hard buckets even slightly, memory might be a side-show compared with core language competence. Obvious interpretation: retrieval does not help at this scale. Alternatives: the datastore is too small; the hidden states were never trained to be retrieval keys; post-hoc kNN and trained ARMT are fundamentally different interventions. Resolve with trained ARMT after `rd12 + control`, not before. Lean: do not move ARMT ahead of collapse repair and control; trained memory is still the right path. Confidence: `8/10`.
- **BayesianWrite is still useful, but `lambda` has not earned the halting role.** For: the writer is stable, bounded, and warm-start compatible; it provides recurrent inertia and prevents raw residual chaos. Against: `lambda` spikes with the final-pass cliff and currently behaves more like deferred budget dumping than calibrated certainty; monotone precision also cannot express contradiction or forgetting. Obvious interpretation: BayesianWrite is broken. Alternatives: the write rule is fine but halting is wrong; `lambda` is learning whatever the loss makes useful, not “confidence”; conflict-aware decay may be the missing piece, not writer replacement. Resolve by correlating `lambda`, entropy, predicted gain, and correctness after `rd12`, plus a small conflict-aware writer canary. Lean: keep BayesianWrite now, demote `lambda` to write inertia, and move halting to entropy/gain. Confidence: `8/10`.
- **Routing is necessary; pheromone is not.** For: long-range communication is still needed, and a local+top-k router is affordable on one GPU. Against: the current system still applies nearly uniform processing, and pheromone is extra machinery with weak evidence and no clean ABI story. Obvious interpretation: routing failed. Alternatives: routing is starved because early passes are dead; routing will only pay once the controller can choose retrieve/verify modes; the failure is precise-memory absence, not routing absence. Resolve with a pheromone ablation and then a controller-guided routing sweep. Lean: keep the local router short term, delete pheromone in the next simplification step, and let the controller decide when retrieval is worth paying for. Confidence: `7/10`.
- **Random-depth is the correct P0 intervention, but it is not automatically the whole fix.** For: it directly changes gradient allocation, it is warm-start compatible, and a live GPU smoke test on the drafted path worked at depths `1/3/6/12`, with peak training-like memory scaling from about `2.15GB` at `D=1` to `9.61GB` at `D=12` for `bs=4, seq=512`. Against: `rd12` alone may create a few decode breakpoints rather than a smooth entropy descent, because the shared block still lacks pass identity. Obvious interpretation: `rd12` will solve collapse. Alternatives: it may only break one cliff into several smaller cliffs; it may preserve D=12 quality but not enable halting; it may expose the need for pass conditioning immediately. Resolve with entropy-curve diagnostics, not only BPT. Lean: branch now, run `3K` as phase 1, and extend only if the entropy curve is improving. Confidence: `9/10`.
- **The `rd12` trainer is close to launch, but metric integrity needs one more bar.** For: warm-start loading is correct, variable-depth forward works on synthetic and real data, and evaluation correctly uses full depth for apples-to-apples BPT. Against: the per-depth eval currently samples different test batches at each depth, so `late_pct` is noisier than necessary, and resume semantics are incomplete because the trainer saves optimizer/depth counts but does not reload them on resume. Obvious interpretation: this is fine because it is “just diagnostics.” Alternatives: noisy diagnostics will miscall success, which is exactly how this project previously wasted time; crash-resume mismatch matters during a 3K probe because the whole point is clean causality. Resolve by caching one held-out batchset across all depths and restoring optimizer/depth counts on true resume. Lean: training can start after those are fixed; they are medium-severity, not thesis-level blockers. Confidence: `8/10`.
- **The current tokenizer/front-end should stay for the warm-start line, even though it is not the endgame.** For: GPT-2 vocab and current embeddings preserve the only compressed knowledge the project has right now. Against: embeddings plus learned positions absorb an absurd share of the parameter budget, and rare-token behavior is still weak. Resolve with a later clean-line canary using `32K BPE + factorized embeddings + RoPE` only after the recurrence/memory gate passes. Lean: freeze the front-end in the warm-start line; change it only in the clean line. Confidence: `9/10`.
- **Recurrence at 68M is still a live hypothesis, not a solved verdict.** For: truncation and late-pass contribution prove there is recurrent value in the current scaffold. Against: the scaffold is presently a one-pass decoder wrapped in eleven latent drifts, and a matched-compute single-pass model may still win. Resolve with the decisive gate on a simplified family after `rd12`, control, and memory exist. Lean: do not kill recurrence now, but do not declare victory before the matched-compute gate either. Confidence: `7/10`.
- **Multi-source learning is the biggest untapped lever, but not the first lever to pull.** For: with one GPU and unlimited free teachers, representation alignment and module-local distillation are the clearest path to Outcome 4. Against: if the architecture is still changing every round, multi-teacher signals will blur attribution and may just inject noise. Resolve with narrow module-local canaries after collapse repair, not with whole-model multi-teacher training immediately. Lean: make teacher absorption a design constraint now, an execution priority immediately after `rd12` stabilizes. Confidence: `7/10`.

5. **What would raise my confidence**
- **Outcome 1:** `rd12` plus control makes the entropy curve graded across passes, and memory-only additions materially improve numbers/entities plus generation coherence, not just recall probes.
- **Outcome 2:** a memory-only or controller-only swap fixes a narrow failure class without degrading general text quality.
- **Outcome 3:** one alternative memory backend plugs in behind the same controller ABI and composes cleanly with the base model.
- **Outcome 4:** a module-local teacher signal gets the same held-out quality in meaningfully fewer tokens, or improves a target capability without whole-model retraining.
- **Outcome 5:** average depth drops below the current fixed-depth equivalent with negligible quality loss, and at least a large easy-token subset reaches low entropy by pass `<=3`.

6. **What would lower my confidence**
- **Outcome 1:** `rd12` preserves `D=12` quality but leaves early passes useless, or trained memory improves recall buckets without improving generation or downstream behavior.
- **Outcome 2:** every improvement still requires whole-model retraining and no module-local intervention isolates anything cleanly.
- **Outcome 3:** the controller/memory ABI keeps changing every round, making module composition unstable.
- **Outcome 4:** teacher signals act like noise, or the warm-start line cannot beat a simple clean restart in tokens-to-quality once architecture changes accumulate.
- **Outcome 5:** the entropy curve remains flat until the last active pass, or memory/control overhead erases the savings from adaptive depth.

7. **Design proposal**
- **Near-term architecture:** keep one shared recurrent content core, but move differentiation into cheap explicit control. This protects Outcome 1 by centralizing content capacity and protects Outcomes 2 and 3 by making control and memory the modular surfaces.
- **Control ABI:** use a 4-mode controller `compose / route / retrieve / verify` with top-2 routing and pass conditioning. Hard tokens need different computation modes; cheap control is the smallest change that can create that diversity without turning the model back into a monolith.
- **Memory spectrum:** keep the scratchpad as workspace, add a zero-param episodic exact buffer, and add ARMT as trained associative memory with forgetting from the first implementation. This directly targets the knowledge gap while preserving a clean memory-backend ABI for Outcomes 2 and 3.
- **Writer and halting:** keep BayesianWrite for bounded state updates, but stop treating `lambda` as the final halting signal. Train with random depth now; later infer with entropy/gain halting once the curve is smooth.
- **Training order:** `rd12` now, pass conditioning if the curve stays cliffed, then 4-mode control, then ARMT plus episodic memory, then shared-core transfer, then the decisive recurrence gate. This is the cheapest sequence that tests one causal claim at a time.
- **Front-end policy:** do not touch tokenizer or embeddings in the warm-start line. If the recurrent architecture survives the gate, then restart the clean line with `32K BPE + factorized embeddings + RoPE`.
- **Data-efficiency policy:** once collapse is repaired, start with module-local teacher absorption, not whole-model imitation: embedding alignment, controller supervision, and memory-write targets are the highest-leverage first teachers.

**Immediate decisions**
- Do **not** move ARMT ahead of `rd12` and pass conditioning; current representations are not retrieval-ready.
- Do **not** repeat large kNN studies before `rd12`; one bigger run is enough as a ceiling probe, not as a gate.
- Branch to `rd12` **now**; start with `3K`, not `5K` upfront, and extend only if the diagnostics are improving.
- Use **logit entropy** as the primary future halting signal; use disagreement as a secondary diagnostic, not the controller.
- Do **not** hardcode pass roles yet; first try soft pass conditioning, because the last-pass cliff is likely learned from supervision, not an inherent need for a special final operation.
- Add a fourth pre-training gate called **measurement integrity**; this project already paid too much for misleading metrics.
- Use **`L_probe` slope plus entropy-curve shape**, not a universal absolute threshold, as the trigger for forced branching.
