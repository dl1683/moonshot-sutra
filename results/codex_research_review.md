**Verdict**

1. **Stage-superposition is not genuinely novel in the strong sense.** In code, it is a recurrent LM with per-token stage probabilities over shared stage MLPs, plus routing and memory, not a new computational class. The implementation in [sutra_v05_ssm.py:108](/C:/Users/devan/OneDrive/Desktop/Projects/AI%20Moonshots/moonshot-sutra/code/sutra_v05_ssm.py#L108), [sutra_v05_ssm.py:207](/C:/Users/devan/OneDrive/Desktop/Projects/AI%20Moonshots/moonshot-sutra/code/sutra_v05_ssm.py#L207), and [sutra_v05_ssm.py:211](/C:/Users/devan/OneDrive/Desktop/Projects/AI%20Moonshots/moonshot-sutra/code/sutra_v05_ssm.py#L211) puts it closer to looped/recurrent/adaptive-compute models than to a paradigm break. The repo itself has a `3/10` novelty rating in [RESEARCH.md:1126](/C:/Users/devan/OneDrive/Desktop/Projects/AI%20Moonshots/moonshot-sutra/research/RESEARCH.md#L1126). Also, the file header says “real verify->reroute loop,” but the forward pass says “No verify/reroute yet” in [sutra_v05_ssm.py:14](/C:/Users/devan/OneDrive/Desktop/Projects/AI%20Moonshots/moonshot-sutra/code/sutra_v05_ssm.py#L14) and [sutra_v05_ssm.py:266](/C:/Users/devan/OneDrive/Desktop/Projects/AI%20Moonshots/moonshot-sutra/code/sutra_v05_ssm.py#L266). That gap matters.

2. **“Intelligence = Geometry” is not defensible as stated.** The defensible version is: geometry of representations and communication graphs matters for efficiency. The repo’s own evidence says the opposite of the strong thesis at current scale: “geometry overhauls” are rejected while simple shared state, switching, Peri-LN, and Grokfast work in [README.md:239](/C:/Users/devan/OneDrive/Desktop/Projects/AI%20Moonshots/moonshot-sutra/README.md#L239) and [RESEARCH.md:3659](/C:/Users/devan/OneDrive/Desktop/Projects/AI%20Moonshots/moonshot-sutra/research/RESEARCH.md#L3659). It is disproved if geometry-heavy changes keep losing to simpler training/data/normalization changes at matched compute.

3. **Telescopic memory is probably the right next step.** It attacks a real bottleneck: the current “sparse” router still forms a full `N x N` score matrix in [sutra_v05_ssm.py:207](/C:/Users/devan/OneDrive/Desktop/Projects/AI%20Moonshots/moonshot-sutra/code/sutra_v05_ssm.py#L207) and [sutra_v05_ssm.py:211](/C:/Users/devan/OneDrive/Desktop/Projects/AI%20Moonshots/moonshot-sutra/code/sutra_v05_ssm.py#L211), while the scratchpad is gist-only working memory in [scratchpad.py](/C:/Users/devan/OneDrive/Desktop/Projects/AI%20Moonshots/moonshot-sutra/code/scratchpad.py). But it should be treated as an engineering memory hierarchy, not a proof of the grand theory.

4. **Will killed mechanisms revive at scale? Some might, most probably won’t.** The repo has real evidence that small-scale Chrome can get the sign wrong: recurrence-depth conclusions flipped, and scratchpad benefit grew at scale. But “retest at scale” is being used too generously in [RESEARCH.md:2339](/C:/Users/devan/OneDrive/Desktop/Projects/AI%20Moonshots/moonshot-sutra/research/RESEARCH.md#L2339) and [SCRATCHPAD.md:65](/C:/Users/devan/OneDrive/Desktop/Projects/AI%20Moonshots/moonshot-sutra/research/SCRATCHPAD.md#L65). My prior is low: retest only mechanisms that solve a diagnosed bottleneck. Complex embeddings and CfC look more like recurring complexity taxes than sleeping giants.

5. **The 15-domain sweep is mostly hypothesis fuel, not evidence.** It may help generate ideas, but right now it is theory surplus. Worse, the repo points to `results/master_research_synthesis.md` in [README.md:331](/C:/Users/devan/OneDrive/Desktop/Projects/AI%20Moonshots/moonshot-sutra/README.md#L331), and that canonical artifact is not present. A missing synthesis artifact undercuts the claim that the sweep is an auditable asset rather than narrative.

6. **Chrome is still useful, but only as a bug-finder and cheap filter.** It clearly helped catch leakage, instability, and some dead ends. It is not reliable for predicting production-scale winners after the dim=128 false positives. The methodology should be downgraded from “evidence of direction” to “early triage,” with a hard ladder: tiny probe, scale proxy, production.

7. **Competitive position vs SmolLM2/Pythia: not competitive yet.** The README target “Competitive with SmolLM2-135M” in [README.md:284](/C:/Users/devan/OneDrive/Desktop/Projects/AI%20Moonshots/moonshot-sutra/README.md#L284) is aspiration, not status. The repo’s earlier “2.5x better than Pythia” claim was explicitly invalidated in [RESEARCH.md:1598](/C:/Users/devan/OneDrive/Desktop/Projects/AI%20Moonshots/moonshot-sutra/research/RESEARCH.md#L1598). There are no serious standard benchmark results yet. Honest position: interesting architecture prototype, well behind serious small-model baselines.

8. **Path to practical intelligence vs forever research:** both are possible, but the current trajectory is closer to forever-research unless the project narrows. There is a plausible practical path as an efficient recurrent LM for long-context recall/reasoning on limited hardware. There is not yet evidence for a general new intelligence paradigm.

**Blunt bottom line**

The real thing here is not “geometry” and not “stage-superposition” as a manifesto. The real thing is a possibly useful recipe: recurrent latent refinement + content-conditioned routing + cheap shared memory. If you focus there, benchmark hard, and stop inflating the theory surface area, this could become a credible efficient-model project. If not, it stays intellectually interesting and practically noncompetitive.
