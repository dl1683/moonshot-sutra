1. **Assumption: fixed `12` recurrent passes is the right depth.**  
For: the current model genuinely uses late recurrence. Truncation at `4/6/8/10` passes leaves BPT in the `18-21` range, while full `12` passes reaches `7.59`, so late compute is real work, not decorative looping.  
Against: the same model shows near-fixed-point collapse through passes `0-9`, with cosine `0.93-0.997`, then a near-orthogonal rewrite at pass `11`. That means the model is not using `12` passes as `12` useful refinements; it is deferring work.  
Interpretations: the obvious reading is “deep recurrence matters.” Alternative reading 1 is “deep recurrence only matters because the training recipe starves early passes.” Alternative reading 2 is “the model is compensating for a weak shared core and huge embedding tax by postponing computation.” Alternative reading 3 is “the scratchpad/routing path only becomes effective late, so the current pass count is masking a coordination problem.”  
Resolve with: same-core `D=1` vs random-depth `D>1` at matched average compute, then `4/8/12` after anti-collapse repair.  
Lean: keep recurrence alive, reject fixed `12` as a design truth. Confidence `8/10`.

2. **Assumption: pure weight sharing across passes is the right parameter strategy.**  
For: on one laptop GPU, aggressive sharing is the only reason recurrence is affordable. Sharing also fits the infrastructure vision because a strong reusable core is easier to improve than many independent blocks.  
Against: the recursive-model survey since Round 1 is nearly unanimous that sub-`200M` successful recursive LMs use pass identity. Only a much larger `3.5B` model gets away with pure sharing. Sutra’s collapse data matches the failure mode that this literature predicts.  
Interpretations: the obvious reading is “no pass identity caused collapse.” Alternative reading 1 is “pass identity is necessary but not sufficient because `L_final` still dominates `97-99%` of gradient at every pass.” Alternative reading 2 is “pure sharing could work if depth were random and short, but not under fixed deep loops.”  
Resolve with: `none` vs `adaLN` vs low-rank depth adapters on the same shared-core canary, with random-depth held constant.  
Lean: keep a mostly shared core, add lightweight pass identity. Confidence `9/10`.

3. **Assumption: the `7`-stage decomposition is the right abstraction.**  
For: stage-superposition generated real positive evidence earlier in the project. At small scale, prose and code followed different stage paths, and stage utilization did diversify. That is one of the best pieces of evidence that Sutra can, in principle, allocate different kinds of computation to different content.  
Against: the current `v0.6.0a` implementation does not realize a true stage ABI. `stage_out` is blended back into one hidden state, stage contracts are not externally stable, and `Segment` and `Address` are mostly dead because tokenization already solved their intended jobs.  
Interpretations: the obvious reading is “stage decomposition is partially validated.” Alternative reading 1 is “the semantic idea is right, but the current heavy realization is wrong.” Alternative reading 2 is “older stage-dynamics evidence came from earlier versions and may not survive the dense-12 regime.”  
Resolve with: compare current `7`-bank design to a single shared content block with lightweight stage-conditioned control, then test whether failures are easier to localize and improve in the lighter system.  
Lean: keep named processing roles, not `7` heavyweight transform banks. Confidence `9/10`.

4. **Assumption: the current `StageBank` mechanism is the right way to implement stage specialization.**  
For: separate stage transforms are the cleanest path to genuinely different computation modes. If routing, memory writing, and verification need different functions, independent transforms are an understandable implementation.  
Against: the current code does not get real token-conditional cheapness. If any token activates a stage, that stage MLP runs over the whole tensor. More importantly, `StageBank` is `16.5M` parameters, which is `24.2%` of the entire model and roughly `58.8%` of the non-embedding compute core. That is too much scarce budget for a mechanism whose unique benefit is still unproven at `68M`.  
Interpretations: the obvious reading is “the stage bank is wasting parameters.” Alternative reading 1 is “the idea may be right at larger scale, but too fragmented here.” Alternative reading 2 is “the control semantics matter, but the data transform should be shared.”  
Resolve with: same-router/same-writer/same-scratchpad comparison between `7` banks and `1` shared SwiGLU block with `pi`-conditioned modulation.  
Lean: replace heavy stage banks with one strong shared block plus lightweight control. Confidence `9/10`.

5. **Assumption: explicit exact-memory support is optional.**  
For rejecting that assumption: this is one of the strongest externally and internally validated pieces of the architecture. The scratchpad gave `+10.2%` BPT in earlier Chrome work and a larger production-scale training-speed benefit. External research now says recurrent/SSM-style systems show an `82%` exact-recall gap against attention and do not close it by scaling alone; they close it with explicit memory. LAMBADA near zero is consistent with that gap.  
Against overclaiming the current implementation: the present scratchpad is only `8` shared slots with soft read/write. That is a crude global workspace, not yet a principled associative memory. Its gains may come from giving the model shared state, not from robust exact recall.  
Interpretations: the obvious reading is “memory is mandatory.” Alternative reading 1 is “the side path is mandatory, but the current scratchpad is only a first draft.” Alternative reading 2 is “the benefit is mostly global coordination, not true exact retrieval.”  
Resolve with: read-disable, write-disable, remove, and ARMT-lite comparisons on exact-recall tasks plus held-out LM slices.  
Lean: explicit memory side-path is non-negotiable; current scratchpad stays for the next gate, but its implementation should remain on trial. Confidence `9/10` on the need for explicit memory, `6/10` on the current scratchpad being the final form.

6. **Assumption: `BayesianWrite` and `lambda` are the right story for state updates and future halting.**  
For: the writer is load-bearing in practice. Gain clamping fixed a real NaN problem, and some bounded write mechanism is clearly needed to stop the recurrent state from becoming a pure residual soup.  
Against: the strong semantic story around `lambda` has not held up. Earlier production probes already showed `lambda` failing as a halting/correctness signal, and the collapse probe now shows a pathological final-pass spike from `1.89` to `4.60`, which looks more like deferred-budget dumping than calibrated certainty. Monotone precision also makes revision under contradiction difficult.  
Interpretations: the obvious reading is “lambda-as-confidence is wrong.” Alternative reading 1 is “the writer is useful, but the interpretation of its scalar is wrong.” Alternative reading 2 is “writer path is not the root cause of collapse; gradient structure is.” Alternative reading 3 is “conflict-aware writing may be useful later, but is not the first anti-collapse lever.”  
Resolve with: current bounded writer vs simpler gated residual writer vs conflict-aware writer on the same simplified recurrent core, after random-depth repair.  
Lean: keep a bounded writer, demote the strong `lambda` halting story, and do not make conflict-aware revision the next big bet. Confidence `7/10`.

7. **Assumption: the current router and pheromone path are the right cross-position communication design.**  
For: routing is clearly necessary. Earlier code-vs-prose transition evidence suggests some content types need extra routing. The late-pass gains may be carried partly by routing plus memory, which means communication is not decorative.  
Against: the current router still builds a full score matrix before sparsification, so the stated efficiency story is not yet true in implementation. Pheromone is even weaker: it adds positive feedback into a system already showing attractor collapse.  
Interpretations: the obvious reading is “keep routing, kill pheromone.” Alternative reading 1 is “late-pass gains may currently depend on pheromone traces more than expected, so removal should be tested only after metric cleanup.” Alternative reading 2 is “the real issue is not pheromone itself but the lack of a low-dimensional control/address channel.”  
Resolve with: simplified-core canary with router kept constant and pheromone toggled, then a second probe splitting content from routing/address features.  
Lean: routing stays; pheromone should be removed unless a clean ablation surprises us. Confidence `8/10` on routing, `7/10` on removing pheromone.

Against the full recipe: the current auxiliary is still a biased sampled-CE signal built from final-pass negatives, and the main failure mode remains that `L_final` dominates gradients at all passes. The recipe trains the model to defer work because early passes almost never get to behave as a final pass.  
Interpretations: the obvious reading is “sampled `L_step` works despite being theoretically wrong.” Alternative reading 1 is “it works only because it is weak, not because it is correct.” Alternative reading 2 is “a soft detached-final consistency loss or residual-prediction objective may beat it once implemented carefully.”

**5. What Would Raise Confidence Per Outcome**

- **Outcome 1:** A same-core `D>1` model beating same-core `D=1` on compute-normalized BPT, generation quality, and exact recall would raise confidence sharply. So would a clear reduction in repetition without decode hacks by `20K+` steps.
- **Outcome 2:** One module upgrade improving a targeted behavior while leaving the rest of the system stable would raise confidence materially. A successful memory-module or controller-module swap would be the cleanest proof.
- **Outcome 3:** A function-preserving version transition where a new module is born at zero influence, trained locally, then composed back into the base model would raise confidence. A small external-style plug-in improvement that works without full retraining would raise it more.
- **Outcome 4:** A single narrow teacher or representation-alignment intervention that reaches a target BPT in fewer tokens than the no-teacher control would raise confidence the most. Evidence that factorized embeddings free large core capacity without wrecking lexical competence would also help.
- **Outcome 5:** Random-depth training that lowers average effective depth without quality loss would raise confidence immediately. If post-fix `ce_spread` still predicts future gain well, that would also validate the path to real halting.

**6. What Would Lower Confidence Per Outcome**

- **Outcome 1:** If the simplified same-core recurrence gate shows `D>1` does not beat `D=1`, confidence should fall sharply. Continued repetition and benchmark stagnation after reaching roughly Chinchilla-scale consumed tokens for this size would lower it further.
- **Outcome 2:** If simplification reveals that almost every meaningful gain lives inside one opaque shared block and cannot be isolated, confidence drops. If improvements still require full retraining every time, the outcome is largely failing.
- **Outcome 3:** If version changes keep breaking checkpoint continuity or require destructive migrations too often, confidence drops. If no stable module interface can be frozen after the next gate, the Linux-like story remains rhetorical.
- **Outcome 4:** If every teacher-based or representation-based intervention either hurts or only matches feeding more raw text, confidence drops. If the architecture never becomes meaningfully more efficient per token than small dense competitors, this outcome is not being served.
- **Outcome 5:** If recurrence only works when nearly all tokens consume near-max depth, confidence drops. If exact recall requires communication so dense that it erases any compute advantage, the efficiency thesis weakens substantially.

**7. Design Proposal**

Do not freeze the full multiscale asynchronous belief graph yet. Freeze only the next architecture question: a **minimal shared-core recurrent exact-memory model** whose sole job is to answer whether recurrence buys anything on this hardware when the known self-inflicted inefficiencies are removed.

- **Use one strong shared content block, not a `7`-FFN stage bank.** This concentrates scarce non-embedding capacity into one repeatedly trained function, which is the most evidence-backed path to better intelligence at `68M`. This serves Outcome 1 directly. It does not have to kill Outcomes 2 and 3, because modularity should move out of duplicated FFNs and into explicit surrounding modules.
- **Keep explicit modules around the shared core: `Router`, `Memory`, and `Control/Verify`.** This is how simplification preserves improvability and democratization. A researcher should be able to improve memory or routing without rewriting the core. The next ABI should therefore be module-level, not “seven hidden MLPs with names.”
- **Add lightweight pass identity.** Use `adaLN` or a low-rank depth adapter so the shared core can behave differently at different passes. Published recursive models below `200M` almost all do this, and Sutra’s collapse profile is the exact failure that pass identity is meant to address.
- **Make random-depth training mandatory.** This is the main anti-collapse mechanism because it changes the gradient structure, not just the representational capacity. For the next gate, the training distribution should be short-loop by default, not fixed `12` by default. This choice serves both Outcome 1 and Outcome 5.
- **Keep the memory side path.** For the immediate gate, keep the current scratchpad because it is the strongest positive signal in the repo. For the first post-gate upgrade, move toward associative or product-key memory so recall is explicit rather than merely “shared workspace-like.” This choice serves Outcome 1 and preserves a clear module boundary for Outcomes 2 and 3.
- **Keep a bounded writer, but stop selling `lambda` as the halting mechanism.** The writer is useful; the semantics around its scalar are not yet trustworthy. Halting should eventually be tied to calibrated gain or disagreement, not monotone precision alone.
- **Use soft intermediate supervision, never intermediate full-vocab CE.** For the immediate gate, the current weak sampled `L_step` is acceptable because it provides gradient flow without collapsing late passes. The first cleaner replacement to test should be detached final-pass consistency or residual prediction. This preserves the recursive structure instead of forcing every pass to become an output head.
- **Design multi-source learning into the interfaces from the start.** Memory/address modules should accept semantic teacher losses from embedding models. The shared core should be able to absorb hidden-state or soft-target guidance from small open LMs on hard-token slices. Domain specialists should attach to separate modules, not to the whole model. This is how the design serves Outcome 4 while also protecting Outcomes 2 and 3.
- **Make function-preserving growth a hard rule.** New modules should enter at zero influence, warm-start with WSD-style schedule resets, and be trainable locally before being merged into the full system. Without that, the model cannot become true infrastructure.
- **Do not commit yet to async halting, multiscale graph topology, tokenizer migration, factorized embeddings, or conflict-aware writing as part of the core gate.** Those are all serious candidates, and factored embeddings are probably the next biggest capacity lever after the gate, but they should not contaminate the answer to the central question.

The design decision should therefore be:

**Build and test a minimal shared-core recurrent exact-memory model with pass identity and random-depth training. Use the same core at `D=1` as the control. If `D>1` does not beat `D=1` on compute-normalized learning and recall, stop treating recurrence as the primary thesis at `68M` and retain only the exact-memory hybrid pieces. If it does win, then the path to a stronger, more modular, more data-efficient Sutra becomes real enough to justify the larger roadmap.**
